# 上下文工程

根据学习路径，第4部分涵盖：**System Prompt → Token 预算 → 项目记忆**

---

## 一、为什么需要上下文工程？

### 1.1 大模型的核心局限

大语言模型（LLM）有三个根本性局限：

| 局限 | 含义 | 解决方案 |
|------|------|----------|
| **无状态** | 不记得上一次对话 | 对话历史持久化 |
| **上下文有限** | 输入 token 有上限（200K） | 压缩策略、记忆系统 |
| **无行动力** | 只能输出文本 | 工具系统 |

Claude Code 的上下文工程就是解决第二个问题的完整方案。

### 1.2 上下文窗口的真实容量

Claude Code 的默认上下文窗口为 200K tokens，但实际可用于对话的空间远小于此：

```
上下文窗口（200K）
├── 系统提示词（~15-25K，缓存后成本低）
├── 工具定义（~10-20K，含 MCP 工具）
├── 用户上下文（CLAUDE.md、git status 等）
├── 输出预留（maxOutputTokens）
│   ├── 默认上限：64K
│   ├── 实际默认：8K（slot-reservation 优化）
│   └── 触顶自动升级：一次 64K 重试
└── 剩余：对话历史空间（随对话增长）
```

## 二、System Prompt 动态组装

### 2.1 三阶段管道

System Prompt 不是一段写死的文本，而是一个 **`string[]` 数组**：

```
getSystemPrompt()          →  string[]       （组装内容）
  ↓
buildEffectiveSystemPrompt() →  SystemPrompt   （选择优先级路径）
  ↓
buildSystemPromptBlocks()  →  TextBlockParam[] （分块 + cache_control 标记）
```

### 2.2 内容结构

| 阶段 | 内容 | 缓存策略 |
|------|------|----------|
| **静态区** | Intro Section、System Rules、Doing Tasks、Actions、Using Tools、Tone & Style、Output Efficiency | 可跨组织缓存（`scope: 'global'`） |
| **BOUNDARY** | `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` | 分界标记（不发送给 API） |
| **动态区** | Session Guidance、Memory、Env Info、Language、Output Style、MCP Instructions | 每次会话不同 |

### 2.3 五级优先级

`buildEffectiveSystemPrompt()` 按以下优先级选择最终 System Prompt：

| 优先级 | 条件 | 行为 |
|--------|------|------|
| **0. Override** | `overrideSystemPrompt` 非空 | 完全替换 |
| **1. Coordinator** | `COORDINATOR_MODE` feature | 使用协调者专用提示词 |
| **2. Agent** | `mainThreadAgentDefinition` 存在 | 替换或追加默认提示词 |
| **3. Custom** | `--system-prompt` 参数指定 | 替换默认提示词 |
| **4. Default** | 无特殊条件 | 使用 `getSystemPrompt()` 完整输出 |

### 2.4 Section 注册表

动态区通过工厂函数注册：

```typescript
// 缓存式 Section：计算一次，/clear 或 /compact 后才重新计算
systemPromptSection('memory', () => loadMemoryPrompt())

// 危险：每轮重新计算，会破坏 Prompt Cache
DANGEROUS_uncachedSystemPromptSection(
  'mcp_instructions',
  () => getMcpInstructionsSection(mcpClients),
  'MCP servers connect/disconnect between turns'  // 必须给出破坏缓存的理由
)
```

**关键源码位置**：
- `src/constants/prompts.ts:444` — `getSystemPrompt()`
- `src/utils/systemPrompt.ts:41` — `buildEffectiveSystemPrompt()`
- `src/constants/systemPromptSections.ts` — Section 注册

## 三、缓存策略：成本优化的核心

### 3.1 为什么是 `string[]` 而非单个 `string`

将 System Prompt 设计为数组是为了 **缓存分块**：

1. Anthropic Prompt Cache 以 **内容块**（TextBlock）为缓存单位
2. 将 System Prompt 拆为多个块，可以让不变的部分获得独立的缓存命中
3. 如果是单个 `string`，任何一个字符变化都会导致整个缓存失效

### 3.2 三种分块模式

`splitSysPromptPrefix()` 根据条件选择分块模式：

#### 模式 1：MCP 工具存在时

```
[attribution header]    → cacheScope: null     （不缓存）
[system prompt prefix]  → cacheScope: 'org'    （组织级缓存）
[everything else]       → cacheScope: 'org'    （组织级缓存）
```

MCP 工具列表在会话中可能变化，破坏了跨组织缓存的基础。

#### 模式 2：Global Cache + Boundary（1P 专用）

```
[attribution header]    → cacheScope: null     （不缓存）
[static content]        → cacheScope: 'global' （全局缓存！跨组织共享）
[dynamic content]       → cacheScope: null     （不缓存）
```

这是缓存效率最高的模式，`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 标记静态区和动态区。

#### 模式 3：默认（3P 提供商）

```
[attribution header]    → cacheScope: null
[system prompt prefix]  → cacheScope: 'org'
[everything else]       → cacheScope: 'org'
```

### 3.3 Provider 系统与缓存

| Provider | 环境变量 | 全局缓存 |
|----------|---------|:---:|
| **Anthropic 直连** | 默认 | ✓ |
| **AWS Bedrock** | `CLAUDE_CODE_USE_BEDROCK=1` | ✗ |
| **Google Vertex** | `CLAUDE_CODE_USE_VERTEX=1` | ✗ |
| **OpenAI 兼容层** | `CLAUDE_CODE_USE_OPENAI=1` | ✗ |
| **Gemini 兼容层** | `CLAUDE_CODE_USE_GEMINI=1` | ✗ |

**关键源码位置**：
- `src/utils/api.ts:321` — `splitSysPromptPrefix()`

## 四、上下文注入：System Context 与 User Context

### 4.1 System Context

```typescript
// src/context.ts:116
export const getSystemContext = memoize(async () => {
  return {
    gitStatus,           // git 分支、状态、最近提交
    cacheBreaker,        // 仅 ant 用户的缓存破坏器
  }
})
```

- 使用 `lodash.memoize` 缓存——**整个会话期间只计算一次**
- Git 状态快照包含 5 个并行命令（branch、defaultBranch、status、log、userName）
- `status` 超过 2000 字符时截断

```typescript
  }
})
```

- **CLAUDE.md 禁用条件**：`CLAUDE_CODE_DISABLE_CLAUDE_MDS` 环境变量，或 `--bare` 模式

## 五、CLAUDE.md：项目级知识注入

### 5.1 多级合并机制

系统自动发现并合并多级 CLAUDE.md：

```
优先级从低到高：
~/.claude/CLAUDE.md              ← 用户全局（个人偏好）
  └── /project/CLAUDE.md         ← 项目根目录（团队共享）
        └── /project/src/CLAUDE.md  ← 子目录（模块特定）
```

加载逻辑从 CWD 向上遍历目录树，合并所有匹配的文件内容。后加载的文件优先级更高。

### 5.2 @include 指令

```markdown
@./relative/path    <!-- 相对路径 -->
@~/home/path        <!-- 用户目录 -->
@/absolute/path     <!-- 绝对路径 -->
```

- 仅在 leaf text nodes 生效（代码块内不处理）
- 循环引用检测
- 静默忽略不存在的文件

### 5.3 使用示例

```markdown
# 项目概述

这是一个 React + TypeScript 的前端项目。

## 开发约定

- 使用函数组件 + Hooks
- 样式使用 Tailwind CSS
- 提交信息遵循 Conventional Commits

## 常用命令

- `npm run dev` - 开发服务器
- `npm run build` - 构建
- `npm run test` - 测试

## 注意事项

- 不要修改 `src/generated/` 下的文件
- 环境变量配置见 `.env.example`
```

**关键源码位置**：`src/utils/claudemd.ts`

## 六、Token 预算管理

### 6.1 上下文窗口计算

`getContextWindowForModel()` 按 5 级优先级解析窗口大小：

1. `CLAUDE_CODE_MAX_CONTEXT_TOKENS` 环境变量覆盖
2. 模型名含 `[1m]` 后缀 → 1M tokens
3. `getModelCapability(model).max_input_tokens`
4. 1M beta header + 支持的模型
5. 兜底：200K

**有效上下文** = 窗口大小 - min(maxOutputTokens, 20K)

### 6.2 Token 计数策略

| 策略 | 速度 | 准确度 | 使用场景 |
|------|------|--------|----------|
| **近似估算** | 毫秒级 | ~90% | 热路径、每轮循环检查 |
| **精确计数** | API 调用 | 100% | 压缩前后对比、warning 判断 |

近似估算规则：
- 普通文本：`length / 4`
- JSON/JSONL：`length / 2`
- 图片/文档：固定 2000 tokens

### 6.3 自动压缩触发阈值

| 常量 | 值 | 含义 |
|------|----|-----|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 自动压缩触发点 |
| `WARNING_THRESHOLD_BUFFER_TOKENS` | 20,000 | 显示警告 |
| `ERROR_THRESHOLD_BUFFER_TOKENS` | 20,000 | 显示错误 |
| `MANUAL_COMPACT_BUFFER_TOKENS` | 3,000 | 手动压缩阻塞上限 |

以 200K 窗口为例：
- **~167K**：warning 提示
- **~180K**：自动压缩触发
- **~197K**：blocking limit

**关键源码位置**：
- `src/utils/context.ts:51` — `getContextWindowForModel()`
- `src/services/compact/autoCompact.ts` — 阈值定义

## 七、压缩策略：三层递进

### 7.1 压缩层级概览

| 层级 | 触发条件 | 实现位置 | 需要 API 调用 |
|------|---------|---------|:---:|
| **MicroCompact** | 单个工具输出过长 | `microCompact.ts` | 否 |
| **Session Memory Compact** | 自动压缩触发 | `sessionMemoryCompact.ts` | 否 |
| **传统 API 摘要** | 手动 `/compact` 或回退 | `compact.ts` | 是 |

### 7.2 MicroCompact — 局部压缩

清除旧工具输出的内容，维护白名单：

```typescript
const COMPACTABLE_TOOLS = new Set([
  'Read',      // 文件读取
  'Bash',      // 命令输出
  'Grep',      // 搜索结果
  'Glob',      // 文件列表
  'WebSearch', // 搜索结果
  'WebFetch',  // 网页内容
  'Edit',      // 编辑输出
  'Write',     // 写入输出
])
```

替换策略：将超过时间窗口的工具输出内容替换为 `[Old tool result content cleared]`。

### 7.3 Session Memory Compact — 无 API 压缩

当 feature flag 启用时，直接使用已提取的 Session Memory 作为摘要：

```typescript
const DEFAULT_SM_COMPACT_CONFIG = {
  minTokens: 10_000,           // 压缩后至少保留 10K token
  minTextBlockMessages: 5,     // 至少保留 5 条文本消息
  maxTokens: 40_000,           // 最多保留 40K token
}
```

### 7.4 传统 API 摘要压缩

当 SM 压缩不可用时，调用 AI 模型生成对话摘要。

压缩后重新注入上下文（50K token 预算）：
- 恢复最近读取的文件内容（最多 5 个文件）
- 恢复已激活的技能指令
- 重新注入 CLAUDE.md 内容

**关键源码位置**：
- `src/services/compact/microCompact.ts`
- `src/services/compact/sessionMemoryCompact.ts`
- `src/services/compact/compact.ts`

## 八、CompactBoundary：压缩边界标记

### 8.1 Boundary 消息

每次压缩后，系统在消息流中插入一条 `SystemCompactBoundaryMessage`：

```typescript
type SystemCompactBoundaryMessage = {
  type: 'system'
  message: {
    type: 'compact_boundary'
    compactMetadata: {
      compactType: 'auto' | 'manual' | 'micro'
      preCompactTokenCount: number
      lastUserMessageUuid: string
    }
  }
}
```

后续操作只处理**最后一条 boundary 之后**的消息。

### 8.2 工具对完整性保护

API 要求每个 `tool_result` 都有对应的 `tool_use`。`adjustIndexToPreserveAPIInvariants()` 确保压缩不会切断工具对。

## 九、项目记忆系统

### 9.1 目录布局

```
~/.claude/projects/<sanitized-git-root>/memory/
├── MEMORY.md                    ← 入口索引（每次对话加载）
├── user_role.md                 ← 用户记忆
├── feedback_testing.md          ← 反馈记忆
├── project_mobile_release.md    ← 项目记忆
└── reference_linear_ingest.md   ← 参考记忆
```

### 9.2 MEMORY.md 索引

```markdown
- [Title](file.md) — one-line hook
```

双重上限：200 行 AND 25KB。

### 9.3 四类型分类法

| 类型 | 存储内容 | 典型触发 |
|------|---------|---------|
| **user** | 用户角色、偏好、技术背景 | "我是数据科学家" |
| **feedback** | 用户对 AI 行为的纠正 | "别 mock 数据库" |
| **project** | 非代码可推导的项目上下文 | "合并冻结从周四开始" |
| **reference** | 外部系统指针 | "pipeline bugs 在 Linear INGEST" |

### 9.4 记忆文件格式

```markdown

name: {{memory name}}
description: {{one-line description}}
```

## 十、记忆漂移防御

### 10.1 验证机制

记忆可能过期。System Prompt 包含专门 section：

```
A memory that names a specific function, file, or flag is a claim
that it existed *when the memory was written*. It may have been
renamed, removed, or never merged. Before recommending it:

- If the memory names a file path: check the file exists.
- If the memory names a function or flag: grep for it.
```

### 10.2 忽略记忆的严格语义

```
If the user says to *ignore* or *not use* memory:
proceed as if MEMORY.md were empty.
Do not apply remembered facts, cite, compare against,
or mention memory content.
```

## 十二、核心设计原则总结

| 原则 | 说明 |
|------|------|
| **分层缓存** | System Prompt 拆分为数组，静态区可跨组织缓存 |
| **动态组装** | 根据会话状态、工具列表、MCP 连接动态构建 |
| **渐进压缩** | MicroCompact → Session Memory → API 摘要三层递进 |
| **智能召回** | Sonnet 侧查询筛选相关记忆，而非全量加载 |
| **漂移防御** | 验证记忆内容，忽略记忆时严格清空 |

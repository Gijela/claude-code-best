# 对话引擎

根据学习路径，第3部分涵盖：**Agentic Loop → 流式响应 → 多轮对话**

## 一、Agentic Loop：AI 自主循环的核心机制

### 1.1 什么是 Agentic Loop

传统聊天机器人是"一问一答"模式，而 Claude Code 采用 **Agentic Loop（智能体循环）**：用户提出一个需求，AI 可能连续执行十几步操作才给出最终结果。

核心实现在 `src/query.ts` 的 `queryLoop()` 函数（第 241 行），它是一个 `while(true)` 无限循环，每次迭代代表一次 **"思考→行动→观察"（ReAct）** 周期。

### 1.2 循环的四阶段结构

```
┌─────────────────────────────────────────────────────────────┐
│                    Agentic Loop 迭代                        │
├─────────────────────────────────────────────────────────────┤
│  ① 上下文预处理管道                                          │
│     applyToolResultBudget → snipCompact → microcompact      │
│     → contextCollapse → autocompact                         │
├─────────────────────────────────────────────────────────────┤
│  ② 流式 API 调用                                             │
│     deps.callModel() → AsyncGenerator<StreamEvent>          │
│     收集 assistantMessages[]、toolUseBlocks[]               │
├─────────────────────────────────────────────────────────────┤
│  ③ 工具执行                                                  │
│     StreamingToolExecutor（并行）或 runTools（串行）         │
│     → toolResults[]                                         │
├─────────────────────────────────────────────────────────────┤
│  ④ 终止/继续判定                                             │
│     needsFollowUp ? continue : return { reason }            │
└─────────────────────────────────────────────────────────────┘
```

**关键源码位置**：`src/query.ts:307` `while(true)` 循环

### 1.3 上下文预处理管道

在调用 API 之前，依次执行 5 个压缩/优化步骤：

| 步骤 | 函数 | 作用 |
|------|------|------|
| **工具结果预算截断** | `applyToolResultBudget()` | 按 `maxResultSizeChars` 截断工具输出 |
| **历史 Snip 压缩** | `snipCompactIfNeeded()` | HISTORY_SNIP feature，按规则压缩历史 |
| **微压缩** | `microcompact()` | 工具结果摘要化，减少 token |
| **上下文折叠** | `applyCollapsesIfNeeded()` | CONTEXT_COLLAPSE feature |
| **自动压缩** | `autocompact()` | 超出阈值时触发完整压缩 |

每个步骤的输出是下一步的输入，形成串行管道。`snipTokensFreed` 会传递给 autocompact 的阈值计算，避免重复压缩。

### 1.4 流式 API 调用

`deps.callModel()` 发起流式请求（第 659 行），返回 AsyncGenerator：

```typescript
for await (const message of deps.callModel({
  messages: prependUserContext(messagesForQuery, userContext),
  systemPrompt: fullSystemPrompt,
  tools: toolUseContext.options.tools,
  ...
})) {
  // 处理流式事件
}
```

流式过程中的关键行为：
- **AssistantMessage** 被收集到 `assistantMessages[]` 数组
- **tool_use 块** 被提取到 `toolUseBlocks[]`，设置 `needsFollowUp = true`
- **StreamingToolExecutor** 在流式过程中就开始并行执行工具（不等流结束）
- 可恢复的错误（prompt-too-long、max-output-tokens）被**暂扣**（withheld）

### 1.5 工具执行

两种工具执行器（互斥）：

```typescript
const toolUpdates = streamingToolExecutor
  ? streamingToolExecutor.getRemainingResults()  // 流式：并行执行
  : runTools(toolUseBlocks, assistantMessages, canUseTool, toolUseContext)  // 串行执行
```

工具执行结果通过 `normalizeMessagesForAPI()` 标准化后，与原始消息合并，进入**下一轮循环迭代**。

## 二、流式响应机制

### 2.1 流式优先设计原则

Claude Code 的所有 API 通信都是流式的：
- `deps.callModel()` 返回 `AsyncGenerator`
- 用户看到 AI "逐字打出"回答
- `StreamingToolExecutor` 在流式过程中就开始并行执行工具

### 2.2 流式事件类型

```typescript
type StreamEvent =
  | { type: 'stream_request_start' }
  | { type: 'content_block_delta'; delta: TextDelta | ToolUseDelta }
  | { type: 'content_block_stop'; index: number }
  | { type: 'message_delta'; delta: MessageDelta; usage: Usage }
  | { type: 'message_stop' }
```

### 2.3 API Provider 支持

`src/services/api/claude.ts` 支持 4 种 Provider：

| Provider | 配置方式 |
|----------|---------|
| **Anthropic Direct** | 默认 |
| **AWS Bedrock** | `ANTHROPIC_BEDROCK_BASE_URL` |
| **Google Vertex** | `ANTHROPIC_VERTEX_PROJECT_ID` |
| **Azure** | 自定义 base URL |

### 2.4 错误恢复与重试

`src/services/api/withRetry.ts` 实现智能重试机制：

## 三、多轮对话管理：QueryEngine

### 3.1 单轮 vs 多轮

| 概念 | 范围 | 管理者 |
|------|------|--------|
| **单轮（一次 Agentic Loop）** | `query()` 的一次完整执行 | `src/query.ts` |
| **多轮（一个 Session）** | 跨越数十轮 `submitMessage()` 调用，持续数小时 | `src/QueryEngine.ts` |

`QueryEngine`（`src/QueryEngine.ts:186`）是单轮 Agentic Loop 之上的**会话编排器**。

### 3.2 QueryEngine 内部状态

```
QueryEngine 内部状态
├── mutableMessages: Message[]           ← 完整对话历史，跨 turn 累积
├── readFileState: FileStateCache        ← 已读文件内容缓存
├── totalUsage: NonNullableUsage         ← 累计 token 消耗
├── permissionDenials: SDKPermissionDenial[]  ← 权限拒绝记录
├── discoveredSkillNames: Set<string>    ← 当前 turn 已发现的 skill
└── abortController: AbortController     ← 会话级中断控制
```

### 3.3 核心方法：submitMessage()

每次用户输入一条消息，REPL 或 SDK 调用 `submitMessage()`：

```typescript
// src/QueryEngine.ts:211 — 简化的 submitMessage 流程
async *submitMessage(prompt, options?): AsyncGenerator<SDKMessage> {
  // 1. 清除 turn 级追踪状态
  this.discoveredSkillNames.clear()

  // 2. 解析模型（用户可能中途切换了模型）
  const mainLoopModel = userSpecifiedModel
    ? parseUserSpecifiedModel(userSpecifiedModel)
    : getMainLoopModel()

  // 3. 动态组装 System Prompt
  const { defaultSystemPrompt, userContext, systemContext } =
    await fetchSystemPromptParts({ tools, mainLoopModel, mcpClients })

  // 4. 调用核心 query() 函数
  yield* query({ systemPrompt, messages: this.mutableMessages, ... })
}
```

关键设计：`submitMessage()` 是 `async *Generator`——逐步 yield `SDKMessage`，让调用方能实时展示进度。

## 四、终止与继续条件

### 4.1 七种终止条件

| 终止原因 | 触发位置 | 机制 |
|----------|---------|------|
| **completed** | 第 1360 行 | AI 未发出 tool_use → stop hooks 通过 → 返回 |
| **blocking_limit** | 第 646 行 | Token 超过硬限制 → 生成 PTL 错误消息 → 返回 |
| **aborted_streaming** | 第 1054 行 | 用户中断 → 为未完成 tool_use 生成合成 tool_result → 返回 |
| **model_error** | 第 999 行 | `callModel()` 抛出异常 → 返回 |
| **prompt_too_long** | 第 1178 行 | 413 错误且无法恢复 → 返回 |
| **image_error** | 第 980/1178 行 | 图片尺寸/大小错误 → 返回 |
| **stop_hook_prevented** | 第 1282 行 | Stop hook 返回 `preventContinuation: true` → 返回 |

### 4.2 四种继续条件（恢复路径）

**1. 正常工具循环**
```
needsFollowUp = true → 执行工具 → 新消息追加 → continue
```

**2. max_output_tokens 恢复**
- 首次：将 `maxOutputTokens` 提升到 64K，静默重试
- 后续：注入恢复消息，最多重试 3 次

**3. Prompt-Too-Long 恢复**
- **Context Collapse Drain**：提交已暂存的折叠，释放空间后重试
- **Reactive Compact**：触发即时压缩，生成摘要后重试

**4. Stop Hook 阻塞重试**
- Stop hook 注入阻塞错误消息，强制 AI 重新思考

## 五、状态机：State 对象

每次迭代的状态通过 `State` 类型传递（`src/query.ts:204`）：

```typescript
type State = {
  messages: Message[]                         // 当前对话消息
  toolUseContext: ToolUseContext              // 工具上下文（含权限）
  autoCompactTracking: AutoCompactTrackingState  // 压缩跟踪
  maxOutputTokensRecoveryCount: number        // 输出截断恢复计数
  hasAttemptedReactiveCompact: boolean        // 是否已尝试即时压缩
  maxOutputTokensOverride: number | undefined // 输出 token 上限覆盖
  pendingToolUseSummary: Promise<...> | undefined  // 异步工具摘要
  stopHookActive: boolean | undefined         // Stop hook 是否激活
  turnCount: number                           // 轮次计数
  transition: Continue | undefined            // 上一次继续的原因
}
```

每次 `continue` 都创建新的 State 对象（不可变更新），`transition` 字段记录为什么继续，避免无限循环。

## 六、会话持久化：Transcript

### 6.1 存储路径

```
~/.claude/projects/<project-hash>/<session-id>.jsonl
```

- `project-hash` 由项目目录生成，同一项目的会话归入同一子目录
- 每条记录是一行 JSON（JSONL 格式），支持追加写入
- 读取上限为 50MB，防止超大会话导致 OOM

### 6.2 会话恢复

`--resume` 参数触发的恢复流程：

```
1. 解析 resume 参数：
   ├── UUID 格式 → getTranscriptPathForSession(uuid)
   ├── .jsonl 文件路径 → 直接使用
   └── boolean → 最近一次会话

2. loadTranscriptFromFile(path)
   └── 重建 Message[] 数组

3. 恢复上下文状态：
   ├── restoreCostStateForSession()  ← 恢复累计费用
   └── 如果有 --rewind-files，恢复文件快照

4. 创建 QueryEngine({ initialMessages })
```

## 八、模型热切换

在一个会话中切换模型不会丢失对话历史：

```
/model sonnet → setMainLoopModelOverride('claude-sonnet-4-20250514')
  ↓
下一次 submitMessage():
  ↓
fetchSystemPromptParts({ mainLoopModel: newModel })
  → System Prompt 根据新模型能力重新组装
  ↓
query({ model: newModel, messages: this.mutableMessages })
  → 使用完整历史 + 新模型继续
```

## 九、工具并行执行

### 9.1 StreamingToolExecutor

`src/services/tools/StreamingToolExecutor.ts` 实现流式工具执行：

```typescript
export class StreamingToolExecutor {
  private tools: TrackedTool[] = []

  // 并发控制
  private canExecuteTool(isConcurrencySafe: boolean): boolean {
    const executingTools = this.tools.filter(t => t.status === 'executing')
    return (
      executingTools.length === 0 ||
      (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
    )
  }
}
```

### 9.2 并发安全工具

工具通过 `isConcurrencySafe()` 方法声明是否可以并行执行：
- **可并行**：Read、Glob、Grep 等只读操作
- **需串行**：Write、Edit、Bash 等写操作

## 十、核心设计原则总结

| 原则 | 说明 |
|------|------|
| **流式优先** | 所有 API 通信都是流式，StreamingToolExecutor 并行执行工具 |
| **工具即能力** | 每个 Tool 是结构化类型，动态组装，支持 MCP 扩展 |
| **权限即边界** | 每次工具调用经过双重检查，Plan Mode 切换为只读模式 |
| **上下文即记忆** | System Prompt 动态组装，Auto-compact 自动压缩 |

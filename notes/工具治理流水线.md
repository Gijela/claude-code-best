# Claude Code 工具治理流水线

> **核心问题**：工具怎么安全执行？
>
> **答案**：不是简单的"调用工具"，而是经过**分层注册、权限检查、输出预算、并发控制**的完整流水线。

---

## 一、工具治理流水线怎么运作

### 调用链路：从 AI 发起到结果回传

```
AI 发出 tool_use（name + input）
        ↓
    参数校验（validateInput）
        ↓ 失败 → 返回错误
    权限检查（checkPermissions + 弹窗确认）
        ↓ 拒绝 → 返回拒绝原因
    执行操作（call）
        ↓
    返回 ToolResult
        ↓
    追加到对话历史 → 进入下一轮迭代
```

### 治理环节：5 道关卡

| 关卡 | 作用 | 示例 |
|------|------|------|
| **分层组装** | 按运行时状态动态加载工具 | 条件工具只在满足条件时启用 |
| **权限分级** | 只读自动放行，写入需确认 | Read 免审批，Write 需确认 |
| **命令解析** | 拆分复合命令，逐段检查 | `ls && git push` 分开判定 |
| **输出预算** | 超长结果截断，防止撑爆上下文 | 限制 maxResultSizeChars |
| **并发控制** | 只读工具可并行，有副作用串行 | `isConcurrencySafe()` 判定 |

## 二、对 Claude Code 工程的价值

### 解决了什么问题

| 问题 | 没有治理流水线 | 有治理流水线 |
|------|--------------|------------|
| **API 报错** | 直接崩溃 | 自动重试/降级 |
| **危险命令** | `rm -rf /` 直接执行 | 权限拦截 + 沙箱隔离 |
| **Token 超限** | 对话崩溃 | 自动压缩工具输出 |
| **工具冲突** | 串行阻塞 | 智能并发调度 |
| **外部工具** | 无法扩展 | MCP 标准协议动态注入 |
| **权限管理** | 全开或全关 | 细粒度规则 + 交互确认 |

### 一句话总结

> 工具治理流水线是"AI 的双手"和"安全护栏"的统一——既要让 AI 能做事，又要防止它做坏事。

## 三、与 Prompt 编排的关联

工具治理不是独立模块，而是与 Prompt 编排紧密耦合：

```
┌─────────────────────────────────────────────────────────────┐
│                    System Prompt 组成                        │
├─────────────────────────────────────────────────────────────┤
│ 静态区（可缓存）                                              │
│   ├── 角色定义                                               │
│   ├── 核心规则                                               │
│   └── 通用指令                                               │
├─────────────────────────────────────────────────────────────┤
│ 动态区（每次重建）                                            │
│   ├── 工具列表定义 ←───── getTools() 返回，占 10-20K tokens    │
│   ├── Tool.prompt() ←──── 每个工具的使用指南                   │
│   └── 当前状态（git/日期/CLAUDE.md）                           │
└─────────────────────────────────────────────────────────────┘
```

### 关键耦合点

| 交互点 | 说明 |
|-------|------|
| **Token 预算** | 工具定义占用 10-20K tokens，直接影响对话历史可用空间 |
| **动态组装** | `getTools()` 每次 API 调用都执行，因为工具列表可能变化 |
| **Tool.prompt()** | 工具不只声明 schema，还告诉模型"怎么用我" |
| **缓存策略** | 工具列表变化会导致 System Prompt 缓存失效 |

**设计意图**：治理决定"有哪些工具可用"，编排决定"模型如何知道这些工具"——是一体两面。

## 四、工具分层组装：四层架构

Claude Code 的所有工具按四个层级组装：

```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：固定工具（始终可用）                                  │
├─────────────────────────────────────────────────────────────┤
│ 第二层：条件工具（运行时检查 isEnabled()）                    │
├─────────────────────────────────────────────────────────────┤
│ 第三层：Feature-flag 工具（实验性功能）                       │
├─────────────────────────────────────────────────────────────┤
│ 第四层：MCP 工具（外部服务提供）                              │
└─────────────────────────────────────────────────────────────┘
```

**关键设计**：工具列表在每次 API 调用时组装（而非全局缓存），因为运行时状态可能变化。

## 五、Claude Code 工具全景

### 第一层：固定工具（核心工具）

这些工具始终可用，构成 AI 的基础能力：

| 分类 | 工具 | 作用 |
|------|-----|------|
| **文件操作** | Read | 读取文件（文本/图片/PDF/Notebook） |
| | Edit | 精确字符串替换，需先 Read 过该文件 |
| | Write | 全量写入或创建文件 |
| | NotebookEdit | 编辑 Jupyter Notebook |
| **命令执行** | Bash | 执行 shell 命令 |
| **网络能力** | WebSearch | 搜索互联网 |
| | WebFetch | 抓取网页内容 |
| **对话与规划** | Agent | 派生子 Agent 处理子任务 |
| | AskUserQuestion | 向用户提问获取澄清 |
| | Skill | 执行技能（/斜杠命令） |
| | EnterPlanMode | 进入计划模式（工具集受限为只读） |
| **任务管理** | TaskCreate / TaskUpdate / TaskList / TaskGet | 任务跟踪 |
| **文件搜索** | Glob | 按文件名模式搜索 |
| | Grep | 按文件内容搜索 |
| **调度** | CronCreate / CronDelete / CronList | 定时任务 |
| **工作树** | EnterWorktree / ExitWorktree | Git worktree 隔离 |

---

### 第二层：条件工具

根据运行时状态决定是否启用：

| 条件 | 工具 | 作用 |
|------|-----|------|
| 无内嵌搜索工具时 | Glob, Grep | 文件名/内容搜索 |
| Todo V2 启用时 | Task 系列 | 任务管理 |
| Worktree 模式时 | EnterWorktree / ExitWorktree | 工作树隔离 |
| Agent Swarms 启用时 | Teams 相关工具 | 多 Agent 协作 |
| Windows 平台 | PowerShell | PowerShell 命令执行 |

---

### 第三层：Feature-flag 工具

通过 feature flag 控制的实验性工具：

| Feature Flag | 工具 | 作用 |
|-------------|------|------|
| `KAIROS` | Sleep, SendUserFile 等 | 长驻助手模式 |
| `WEB_BROWSER_TOOL` | WebBrowser | 浏览器控制 |
| `DAEMON` | Daemon 相关 | 后台守护进程 |

---

### 第四层：MCP 工具

由 MCP Server 提供的外部工具，命名规则：

```
mcp__<server_name>__<tool_name>
```

**示例**：
- `mcp__postgres__query` — PostgreSQL 查询
- `mcp__github__create_issue` — GitHub 创建 issue
- `mcp__slack__send_message` — Slack 发消息

**特点**：
- 动态注入，连接 MCP Server 时自动发现
- 支持按 server 前缀整体禁用（blanket deny）

## 六、权限模型：三层防御

```
┌─────────────────────────────────────────────────────────────┐
│ 第一层：应用层权限（settings.json）                           │
│   • allow: ["Read(**)", "Bash(git status:*)"]              │
│   • deny: ["Bash(rm -rf:*)", "Write(.env)"]                │
├─────────────────────────────────────────────────────────────┤
│ 第二层：交互层确认（弹窗）                                     │
│   • 只读操作：自动放行                                        │
│   • 写入操作：弹出确认框，可选择"始终允许"                     │
│   • 危险操作：强制确认 + 风险提示                              │
├─────────────────────────────────────────────────────────────┤
│ 第三层：沙箱隔离（可选）                                       │
│   • worktree 模式：在隔离目录执行                             │
│   • 防止误删主仓库                                            │
└─────────────────────────────────────────────────────────────┘
```

**权限规则语法**：
```
Read(**)              // 所有读取操作
Write(src/**)         // 仅 src 目录写入
Bash(git *)           // git 开头的命令
```

**匹配优先级**：deny > allow > 默认行为
